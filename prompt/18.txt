중요:

- 기존 UI 디자인은 가능한 유지한다.
- 이번 작업은 기상청 단기예보 조회서비스를 서버 측에서 호출하고, 운전 위험 지수 계산에 사용할 기상 위험 가중치를 만드는 단계다.
- 아직 Kakao Local API, Kakao Map, Claude API 연동은 하지 않는다.
- ODsay 연동 흐름은 유지한다.
- Supabase 저장/조회 흐름은 유지한다.
- 아직 회원가입/auth 구현하지 않는다.
- 아직 관리자 페이지, 결제, 알림, 카카오 공유 구현하지 않는다.
- Weather API가 실패하거나 키가 없거나 응답 구조가 예상과 달라도 sampleWeather fallback으로 화면이 깨지지 않게 한다.
- WEATHER_API_KEY는 공공데이터포털 “기상청_단기예보 조회서비스” 일반 인증키/serviceKey를 사용한다.
- WEATHER_API_KEY는 서버 Route Handler에서만 사용하고 클라이언트에 노출하지 않는다.

디자인 관련 주의:
- 이번 작업은 디자인 변경/반영 단계가 아니다.
- 디자인이나 벤치마킹 사이트 반영이 필요한 작업이 다시 오면, 반드시 작업 시작 전에 아래처럼 큰 제목으로 먼저 알려줘.

# 🚨 디자인 변경/벤치마킹 반영 단계입니다

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 사용하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- API 실패 시 sampleAnalysis, sampleRoute, sampleWeather, generateTemplateReport 등 fallback으로 화면이 깨지지 않게 한다.
- “무장애”보다 “이동권 보장”, “이동 편의 동선” 표현을 우선 사용한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

작업 목표:
- 기상청 단기예보 조회서비스를 서버 측에서 호출하는 구조를 만든다.
- 좌표 또는 대전 기본 위치를 기준으로 단기예보 데이터를 조회한다.
- 기상청 원본 응답을 앱 내부 WeatherRisk 타입으로 정규화한다.
- 비, 눈, 강수확률, 강수형태, 풍속, 기온 등 주요 값을 바탕으로 기상 위험 가중치를 계산한다.
- /api/weather Route Handler에서 실제 Weather 조회를 시도하고, 실패 시 sampleWeather fallback을 반환한다.
- /api/analyze에서 가능한 경우 Weather 결과를 운전 위험 지수 계산에 반영한다.
- API key 누락, 네트워크 실패, 응답 오류, 예보 없음 상황에서도 앱이 깨지지 않게 한다.
- 실제 Kakao 좌표 검색은 아직 하지 않고, request body 좌표 또는 대전 기본 좌표를 사용한다.

1. 환경변수
- .env.local.example에 WEATHER_API_KEY가 이미 있는지 확인한다.
- 없으면 추가한다.

WEATHER_API_KEY=

주의:
- WEATHER_API_KEY는 공공데이터포털 “기상청_단기예보 조회서비스” 일반 인증키/serviceKey를 넣는다.
- NEXT_PUBLIC_ 접두사를 붙이지 않는다.
- 클라이언트 컴포넌트에서 사용하지 않는다.
- 서버 Route Handler 또는 서버 유틸에서만 읽는다.
- 응답 JSON이나 console.log에 key를 노출하지 않는다.

2. lib/weather/types.ts
- 기상청 API 원본 응답 타입은 필요한 최소 필드만 정의한다.
- 전체 원본 스펙을 모두 타입화하려고 하지 않는다.
- unknown을 받은 뒤 안전하게 파싱하는 구조를 우선한다.
- any 사용은 피하되, 원본 응답이 복잡하면 최소 범위에서만 사용한다.

권장 타입:
export type WeatherRiskRequest = {
  lat?: number;
  lng?: number;
  baseDateTime?: string;
};

export type WeatherRiskResult =
  | { ok: true; weather: WeatherRisk; source: "KMA" }
  | { ok: false; weather: WeatherRisk; source: "FALLBACK"; reason: string };

export type WeatherCategory = "TMP" | "POP" | "PTY" | "PCP" | "SNO" | "SKY" | "WSD" | string;

필요하면 KmaForecastItem, KmaForecastResponse 같은 내부 타입을 최소 정의한다.

3. lib/weather/convertGrid.ts
- 기상청 단기예보는 위경도 좌표가 아니라 격자 x, y를 사용하므로 변환 함수를 작성한다.
- 위경도 → 기상청 격자 좌표 변환 함수를 만든다.
- 함수명 예시:
export function convertLatLngToGrid(lat: number, lng: number): { nx: number; ny: number }

- 입력이 유효하지 않으면 대전 기본 격자값을 반환하거나 안전한 fallback을 사용한다.
- 대전 기본 좌표/격자값을 상수로 둔다.
- 실제 Kakao 좌표 검색은 하지 않는다.

주의:
- 변환 함수는 순수 함수로 작성한다.
- API 호출을 하지 않는다.
- 테스트 가능하게 만든다.

4. lib/weather/normalizeWeatherForecast.ts
- 기상청 원본 응답을 앱 내부 WeatherRisk 타입으로 변환한다.
- 순수 함수로 작성한다.
- API 호출은 하지 않는다.
- 테스트하기 쉬운 구조로 만든다.

권장 함수:
export function normalizeWeatherForecast(raw: unknown): WeatherRisk | null

정규화 기준:
- TMP: 기온
- POP: 강수확률
- PTY: 강수형태
- PCP: 강수량
- SNO: 적설
- SKY: 하늘상태
- WSD: 풍속
- condition: 사용자에게 보여줄 날씨 요약
- riskScore: 0~100 기준 기상 위험 가중치 산정용 점수
- description: 운전 위험 지수에 참고되는 기상 요인 설명

기상 위험 점수 예시:
- 맑음/강수 없음/약한 바람: 0~20
- 흐림 또는 약한 비 가능성: 20~45
- 비/눈/강한 바람: 45~75
- 강한 비/눈/매우 강한 바람: 75~100

주의:
- 이 점수는 실제 사고 가능성이 아니라 운전 위험 지수에 반영되는 기상 가중치라는 점을 설명한다.
- “사고 확률”, “예측 확률”, “probability” 표현 금지.
- 기상 조건을 과도하게 공포스럽게 표현하지 않는다.
- 원본 응답에 예보가 없거나 파싱 실패 시 null을 반환한다.

5. lib/weather/getWeatherRisk.ts
- WEATHER_API_KEY를 서버 측에서 읽는다.
- 기상청 단기예보 조회서비스를 호출한다.
- 좌표가 들어오면 convertLatLngToGrid로 nx, ny를 계산한다.
- 좌표가 없으면 대전 기본 위치를 사용한다.
- fetch timeout 또는 AbortController를 사용해 너무 오래 대기하지 않게 한다.
- 실패 시 sampleWeather fallback을 반환한다.
- 외부 API 호출 실패를 throw로 전파하지 말고 WeatherRiskResult 형태로 반환한다.

권장 함수:
export async function getWeatherRisk(input?: WeatherRiskRequest): Promise<WeatherRiskResult>

처리해야 할 실패 케이스:
- WEATHER_API_KEY 누락
- 좌표 누락 또는 숫자 아님
- 격자 변환 실패
- fetch 실패
- HTTP 응답 실패
- 기상청 응답 내 오류
- 예보 없음
- 정규화 실패

fallback 시:
- sampleWeather를 반환한다.
- source는 "FALLBACK"으로 둔다.
- reason에 내부 확인용 짧은 이유를 남긴다.
- 사용자-facing 문구에는 과도한 오류 표현을 쓰지 않는다.

주의:
- WEATHER_API_KEY를 응답 JSON에 포함하지 않는다.
- console.log로 key를 출력하지 않는다.
- 클라이언트에서 직접 기상청 API를 호출하지 않는다.

7. app/api/weather/route.ts
- 기존 sampleWeather 반환 route를 실제 Weather 연동 route로 확장한다.
- GET과 POST 중 하나 또는 둘 다 지원할 수 있다.
- 추천:
  - GET /api/weather: 대전 기본 위치 기준
  - POST /api/weather: body에 lat/lng를 받아 조회

POST body 예시:

  "lat": 36.3504,
  "lng": 127.3845
}

응답 예시:
{
  "ok": true,

    "source": "KMA",
    "fallback": false
  }
}

fallback 응답 예시:
{
  "ok": true,
  "mode": "KMA_OR_FALLBACK",
  "data": WeatherRisk,
  "meta": {
    "source": "FALLBACK",
    "fallback": true,

  }
