너는 15년차 Next.js 개발자이자 공공데이터·AI 서비스 실서비스 전환 전문가다.
나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.

현재 상황:
- 이전 작업 1~24는 완료되었다.
- Supabase analysis_logs 저장/조회 구조는 이미 구현되어 있다.
- lib/supabase/analysisLogs.ts, app/api/analyze/route.ts, app/api/result/[id]/route.ts, app/result/[id]/page.tsx 또는 유사 파일이 존재한다.
- /api/analyze는 분석 결과를 저장하고 resultId를 반환하는 구조로 구현되어 있다.
- /api/result/[id]는 Supabase에서 저장된 분석 결과를 조회하는 구조로 구현되어 있다.
- fallback, FallbackFlags, 에러/로딩 처리도 이미 구현되어 있다.
- 이번 작업은 Supabase 저장/조회를 새로 만드는 것이 아니라, 운영 환경에서 실제 DB 저장/조회가 제대로 되는지 확인하고 필요한 경우 최소 수정하는 작업이다.

현재 점검 필요 사항:
- /api/analyze POST 후 resultId가 실제 UUID인지, mock-* 또는 fallback id인지 확인해야 한다.
- Supabase Dashboard의 analysis_logs 테이블에 실제 row가 생성되는지 확인해야 한다.
- /api/result/[id]가 저장된 row를 실제로 조회하는지 확인해야 한다.
- 저장 실패 시에도 화면이 깨지지 않는 persistence/fallback 구조는 유지해야 한다.
- SUPABASE_SERVICE_ROLE_KEY는 서버에서만 사용되어야 한다.

이번 작업은 새롭게 시작하는 실사용 전환 프롬프트 5번이다.
파일명 기준으로는 5.txt 작업이다.

이번 작업명:
Supabase 저장/조회 운영 확인 및 최소 보정

작업 목적:
- /api/analyze가 운영 환경에서 실제 Supabase analysis_logs에 저장하는지 확인한다.
- 저장 성공 시 resultId가 실제 저장된 row의 id와 일치하는지 확인한다.
- /api/result/[id]가 실제 Supabase row를 조회해 결과 페이지에 표시하는지 확인한다.
- 저장/조회 실패 시 fallback으로 화면이 깨지지 않게 유지한다.
- Supabase 저장/조회 구조를 재작성하지 않고, 필요한 경우 최소 수정만 한다.
- 기존 디자인, 레이아웃, UI 스타일은 변경하지 않는다.

절대 하지 말 것:
- Supabase 저장/조회 구조를 처음부터 다시 만들지 마.
- migration schema를 무단으로 변경하지 마.
- analysis_logs 테이블을 새로 만들지 마.
- RLS 정책을 임의로 대폭 변경하지 마.
- 클라이언트에서 service role key를 사용하지 마.
- 클라이언트에서 직접 analysis_logs insert/select를 구현하지 마.
- Kakao Local 연동을 수정하지 마.
- ODsay 연동을 수정하지 마.
- Weather 연동을 수정하지 마.
- Claude 연동을 수정하지 마.
- Kakao Map SDK를 수정하지 마.
- RecentPlaceList localStorage 로직을 수정하지 마.
- 디자인 변경하지 마.
- 레이아웃 변경하지 마.
- 패키지 설치하지 마.
- 대규모 리팩토링하지 마.
- fallback 구조를 제거하지 마.

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 새로 추가하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- fallback은 운영 안정성을 위한 대체 응답으로 유지한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

점검 및 수정 지침:

1. Supabase 환경변수 확인
다음 환경변수가 서버에서 정상적으로 읽히는지 확인한다.
- NEXT_PUBLIC_SUPABASE_URL
- NEXT_PUBLIC_SUPABASE_ANON_KEY
- SUPABASE_SERVICE_ROLE_KEY

주의:
- SUPABASE_SERVICE_ROLE_KEY는 서버 전용이다.
- 클라이언트 컴포넌트, components, app/page.tsx, app/layout.tsx, 브라우저 번들에서 참조하면 안 된다.
- API 응답, console.log, localStorage에 service role key가 노출되면 안 된다.
- /api/health는 boolean 상태만 보여줘야 하며 키 값을 보여주면 안 된다.

2. saveAnalysisLog 동작 확인
lib/supabase/analysisLogs.ts 또는 유사 파일에서 저장 함수가 실제 insert를 수행하는지 확인한다.

확인할 것:
- analysis_logs 테이블에 insert하는가?
- insert 후 id를 반환하는가?
- result.id와 DB row id가 일치하는가?
- result_json 또는 payload jsonb 컬럼에 AnalysisResult 전체가 저장되는가?
- request_json 또는 request 관련 컬럼에 origin/destination/departureTime/ageGroup이 저장되는가?
- risk_score, risk_level, origin/destination 정보가 컬럼에 매핑되는가?
- 저장 실패 시 throw로 API 전체가 깨지지 않고 ok false 또는 persistence.saved false를 반환하는가?

권장 저장 결과 타입:
{
  ok: true,
  id: string
}

실패 시:
{
  ok: false,
  id?: string,
  message: "SAVE_FAILED"
}

주의:
- DB 내부 에러 메시지를 사용자에게 그대로 노출하지 않는다.
- 서버 로그에는 원인 확인이 가능하되 민감정보는 제외한다.

3. /api/analyze 저장 결과 확인
app/api/analyze/route.ts에서 분석 결과 생성 후 saveAnalysisLog를 호출하는지 확인한다.

응답 권장 구조:
{
  "ok": true,
  "data": AnalysisResult,
  "resultId": "uuid",
  "fallbackFlags": { ... },
  "persistence": {
    "saved": true
  }
}

저장 실패 시:
{
  "ok": true,
  "data": AnalysisResult,
  "resultId": "fallback-or-local-id",
  "fallbackFlags": { ... },
  "persistence": {
    "saved": false,
    "reason": "SAVE_FAILED"
  }
}

확인할 것:
- 저장 성공 시 resultId가 mock-* 형식이 아닌 실제 DB id인지 확인한다.
- 저장 성공 시 persistence.saved가 true인지 확인한다.
- 저장 실패 시에도 data는 반환되어 화면이 깨지지 않는지 확인한다.
- mode 값에 MOCK 문자열이 다시 들어가지 않는지 확인한다.
- API 응답에 SUPABASE_SERVICE_ROLE_KEY 또는 내부 에러 전체가 포함되지 않는지 확인한다.

4. /api/result/[id] 조회 동작 확인
app/api/result/[id]/route.ts에서 실제 Supabase select를 수행하는지 확인한다.

확인할 것:
- id param을 받아 analysis_logs에서 select하는가?
- result_json 또는 payload jsonb 컬럼을 AnalysisResult로 반환하는가?
- 없는 id일 때 500으로 깨지지 않는가?
- 없는 id일 때 fallback을 반환하거나 사용자에게 안전한 안내를 제공하는가?
- 조회 성공 시 meta.source 또는 persistence 정보를 통해 DB 조회 성공 여부를 알 수 있는가?
- 응답에 MOCK 문자열이 포함되지 않는가?

권장 응답 구조:
{
  "ok": true,
  "data": AnalysisResult,
  "meta": {
    "source": "SUPABASE",
    "fallback": false
  }
}

없는 id 또는 조회 실패 시:
{
  "ok": true,
  "data": fallbackAnalysis,
  "meta": {
    "source": "FALLBACK",
    "fallback": true,
    "reason": "RESULT_NOT_FOUND"
  }
}

주의:
- 실제 운영에서는 없는 결과를 무조건 성공처럼 보이게 하는 것이 애매할 수 있다.
- 현재 MVP/실서비스 전환 단계에서는 화면 안정성을 우선하되, meta.reason으로 구분 가능하게 한다.

5. result 페이지 연결 확인
app/result/[id]/page.tsx 또는 관련 result fetch 로직을 확인한다.

확인할 것:
- URL의 id를 사용해 /api/result/[id] 또는 서버 함수를 호출하는가?
- 항상 /result/test 또는 fallback만 보여주고 있지 않은가?
- 실제 DB 조회 성공 시 저장된 결과를 표시하는가?
- 로딩/에러 상태에서 화면이 깨지지 않는가?
- 디자인, 카드, 버튼, 레이아웃은 변경하지 않는다.

6. Supabase schema 매핑 확인
supabase/migrations/*.sql 또는 현재 타입을 확인하여 insert/select 컬럼명이 맞는지 확인한다.

확인할 것:
- analysis_logs 테이블명 정확성
- id 타입
- result_json/payload 컬럼명
- request_json 컬럼명
- risk_score/risk_level 컬럼명
- origin/destination 관련 컬럼명
- created_at default 여부
- user_id가 nullable인지, auth 없이 insert 가능한 구조인지

주의:
- migration 변경이 꼭 필요하면 별도 작업으로 분리하고 이번 작업에서는 보고만 한다.
- 단순 컬럼명 불일치로 코드가 실패하는 경우에는 코드의 insert payload를 실제 schema에 맞게 최소 수정한다.

7. RLS 및 권한 점검
회원가입/auth가 없는 서비스이므로 현재 저장은 서버 Route Handler에서 service role로 수행되어야 한다.

확인할 것:
- 클라이언트에서 anon key로 insert하지 않는가?
- 서버에서 service role client를 사용하는가?
- service role key가 서버 전용 파일에만 있는가?
- RLS가 켜져 있어도 service role로 insert/select가 가능한가?

주의:
- RLS 정책을 무리하게 공개 insert 허용으로 바꾸지 않는다.
- anon insert/select를 열지 않는다.
- 서버 Route Handler를 통한 저장/조회 구조를 유지한다.

8. 테스트 작성/보강
실제 Supabase 네트워크 호출에 의존하지 않는다.
Supabase client는 mock한다.

필수 테스트:
- AnalysisResult가 analysis_logs insert payload로 변환된다.
- origin/destination name/address/lat/lng가 payload에 포함된다.
- drivingRisk.score와 level이 payload에 포함된다.
- 저장 성공 시 ok true와 id를 반환한다.
- 저장 실패 시 throw하지 않고 ok false를 반환한다.
- /api/analyze 응답에서 persistence.saved true/false가 구분된다.
- /api/result/[id] 조회 성공 시 source가 SUPABASE로 표시된다.
- 조회 실패 시 fallback으로 안전하게 응답한다.
- service role key가 응답 문자열에 포함되지 않는다.
- 응답 문자열에 금지 표현이 포함되지 않는다.

- 예측 확률

주의:
