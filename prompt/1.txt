너는 15년차 Next.js 개발자이자 공공데이터·AI 서비스 실서비스 전환 전문가다.
나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.

현재 상황:
- 이전 작업 1~24는 완료되었다.
- ODsay, 기상청, Claude, Supabase 저장/조회, Kakao Map, Kakao Local 서버 구조는 이미 구현되어 있다.
- fallback 전체 구조, FallbackFlags, 에러/로딩 처리도 이미 구현되어 있다.
- 테스트 229개, Vercel 배포, 환경변수 보안 분리도 완료된 상태다.
- 따라서 이번부터는 기존 구현을 다시 만들지 않고, 실제 서비스 전환에 필요한 작은 수정만 진행한다.

이번 작업은 새롭게 시작하는 실사용 전환 프롬프트 1번이다.
파일명 기준으로는 1.txt 작업이다.

이번 작업명:
Mock/시연용 표현 운영 정리

작업 목적:
- 사용자 화면과 API 응답에서 “Mock”, “시연용”, “더미”처럼 실제 서비스에 어울리지 않는 표현을 제거한다.
- 기존 디자인, 레이아웃, 색상, 카드 스타일, 버튼 스타일은 변경하지 않는다.
- 이미 구현된 외부 API 연동, Supabase 저장/조회, fallback 구조는 다시 작성하지 않는다.
- 이번 작업은 텍스트와 mode 값 정리 중심의 작은 수정이다.

절대 하지 말 것:
- 디자인 변경하지 마.
- 레이아웃 변경하지 마.
- 컴포넌트 구조를 크게 바꾸지 마.
- Kakao Local 연동을 다시 구현하지 마.
- ODsay 연동을 다시 구현하지 마.
- Weather 연동을 다시 구현하지 마.
- Claude 연동을 다시 구현하지 마.
- Supabase 저장/조회 로직을 다시 구현하지 마.
- fallback 구조를 제거하지 마.
- 회원가입/auth 구현하지 마.
- 관리자 페이지 구현하지 마.
- 결제/알림/카카오 공유 구현하지 마.
- 패키지 설치하지 마.
- 대규모 리팩토링하지 마.

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 새로 추가하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- fallback은 운영 안정성을 위한 대체 응답으로 유지한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

필수 수정 항목:

1. ResultSummary 문구 정리
현재 “시연용 Mock 결과” 뱃지 또는 유사 표현이 있다면 실제 서비스에 어울리는 표현으로 교체한다.

권장 교체:
- “시연용 Mock 결과” → “예시 결과”
또는
- “참고용 결과”
또는
- “보조 데이터 기반 결과”

주의:
- 뱃지의 디자인, 색상, 위치, 크기는 변경하지 않는다.
- 텍스트만 변경한다.
- fallback 상황을 숨기지 말고, 사용자에게 부드럽게 안내하는 표현으로 정리한다.

2. DataSourceCard 문구 정리
현재 “시연용 Mock 또는 Fallback” 같은 표현이 있다면 운영용 표현으로 바꾼다.

권장 교체:
- “시연용 Mock 또는 Fallback” → “예시 또는 보조 데이터”
또는
- “보조 데이터 기반”
또는
- “대체 데이터 사용”

주의:
- fallback이라는 내부 개발 용어를 사용자에게 과도하게 노출하지 않는다.
- 단, 데이터 출처가 실제 API인지 보조 데이터인지 구분은 가능해야 한다.
- 카드 레이아웃과 스타일은 변경하지 않는다.

3. RecentPlaceList 문구 정리
현재 “추천 장소 (시연용)” 또는 유사 표현이 있다면 “추천 장소” 또는 “최근 장소”로 정리한다.

권장 교체:
- “추천 장소 (시연용)” → “추천 장소”
또는
- “최근 장소”

주의:
- RecentPlaceList의 localStorage 실제 동작화는 다음 작업에서 진행한다.
- 이번 작업에서는 텍스트 정리만 한다.
- samplePlaces 하드코딩 구조를 이번 작업에서 바꾸지 않는다.
- 기능 구현을 섞지 않는다.

4. /api/analyze mode 값 정리
API 응답에서 mode 값에 “MOCK”이 포함되어 있다면 운영 전환용 이름으로 바꾼다.

권장 교체:
- “MOCK_WITH_STORAGE” → “ANALYSIS_WITH_STORAGE”
- “MOCK_WITH_STORAGE_FALLBACK” → “ANALYSIS_FALLBACK”
- “MOCK” → “ANALYSIS”
- “MOCK_FALLBACK” → “ANALYSIS_FALLBACK”

주의:
- 클라이언트가 mode 값을 직접 분기하고 있다면 함께 확인하고 깨지지 않게 수정한다.
- 기존 응답 구조 자체는 유지한다.
- resultId, data, fallbackFlags, persistence 구조는 유지한다.
- Supabase 저장 로직은 건드리지 않는다.
- ODsay/Weather/Claude 호출 로직은 건드리지 않는다.

5. API 응답 message 정리
아래 문구가 있으면 운영용 표현으로 변경한다.

교체 예시:
- “Mock 분석 결과를 반환했습니다.” → “분석 결과를 반환했습니다.”
- “Mock 분석 결과를 저장했습니다.” → “분석 결과를 저장했습니다.”
- “시연용 결과입니다.” → “보조 데이터가 일부 사용된 결과입니다.”

주의:
- 사용자에게 내부 구현명을 노출하지 않는다.
- fallback 사용 자체는 fallbackFlags 또는 dataSources에서 확인 가능하게 유지한다.

6. /result/test 노출 문구 정리
/result/test 페이지가 남아 있다면 사용자에게 “시연용” 또는 “Mock”으로 보이지 않게 정리한다.

가능한 처리:
- 페이지 내부 문구를 “예시 결과 확인” 정도로 변경
- 운영 메인 플로우에서 해당 링크가 노출되지 않는지 확인
- 디자인은 변경하지 않는다.

주의:
- /result/test 페이지 삭제는 이번 작업에서 하지 않는다.
- 라우팅 구조 변경하지 않는다.
- 링크 제거가 이미 되어 있다면 추가 작업하지 않는다.

7. README 또는 docs 문구 정리
사용자/운영 설명에 “시연용 Mock”, “더미”, “Mock API” 표현이 남아 있다면 실제 서비스 전환용 표현으로 변경한다.
