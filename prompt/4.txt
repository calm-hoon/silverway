서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.

중요 판단:

- ODsay API 연동 구조는 이미 있으므로 재작성하지 말고 최소 수정한다.

이번 작업은 새롭게 시작하는 실사용 전환 프롬프트 4번이다.

점검 및 수정 지침:

1. ODsay 요청 URL 확인
ODsay 대중교통 길찾기 API 호출 endpoint가 올바른지 확인한다.

일반적인 확인 포인트:
- searchPubTransPathT 또는 현재 프로젝트에서 사용하는 ODsay endpoint가 맞는지
- GET 방식인지 POST 방식인지 현재 코드와 API 요구사항이 맞는지
- query string에 필수 파라미터가 모두 포함되는지
- apiKey 파라미터 이름이 정확한지
- URLSearchParams를 사용할 때 apiKey가 깨지지 않는지

필수 파라미터 확인:
- SX: 출발 경도 lng
- SY: 출발 위도 lat
- EX: 도착 경도 lng
- EY: 도착 위도 lat
- apiKey: ODSAY_API_KEY

중요:
- 좌표 순서를 반드시 확인한다.
- lat/lng를 반대로 넣으면 정상 경로를 찾지 못할 수 있다.
- ODsay는 일반적으로 X=경도, Y=위도 방식이다.
- origin.lng → SX
- origin.lat → SY
- destination.lng → EX
- destination.lat → EY

2. 입력 body 파싱 확인
/api/route가 받는 body 구조를 확인한다.

반드시 지원할 입력 예시:
{
  "origin": {
    "name": "대전광역시청",
    "lat": 36.3504,
    "lng": 127.3845
  },
  "destination": {
    "name": "충남대학교병원",
    "lat": 36.3166,
    "lng": 127.4156
  }
}

확인할 것:
- origin.lat/lng가 number로 들어오는지
- destination.lat/lng가 number로 들어오는지
- string으로 들어오면 number 변환을 하는지
- 누락 시 fallback으로 안전하게 처리하는지
- 좌표가 0, NaN, undefined이면 ODsay 호출하지 않고 명확한 reason을 반환하는지

3. ODsay 원본 응답 처리 개선
현재 reason이 "ODsay error code= msg="로 비어 있다.
ODsay 응답 body를 안전하게 파싱해 원인을 구분한다.

확인할 것:
- HTTP status가 200이 아닌 경우 status를 reason에 반영
- ODsay 응답의 error.code, error.msg 또는 result.error 등 실제 구조를 확인
- JSON parse 실패 시 body 일부를 안전하게 요약
- 응답은 200이지만 경로가 없는 경우 NO_ROUTE_FOUND로 구분
- timeout이면 ODSAY_TIMEOUT으로 구분
- fetch 자체 실패면 ODSAY_FETCH_FAILED로 구분

권장 reason:
- ODSAY_API_KEY_MISSING
- INVALID_COORDINATES
- ODSAY_HTTP_ERROR_STATUS_401
- ODSAY_HTTP_ERROR_STATUS_403
- ODSAY_HTTP_ERROR_STATUS_429
- ODSAY_HTTP_ERROR_STATUS_500
- ODSAY_API_ERROR_CODE_{code}
- ODSAY_EMPTY_RESULT
- ODSAY_NO_ROUTE_FOUND
- ODSAY_TIMEOUT
- ODSAY_PARSE_ERROR
- ODSAY_FETCH_FAILED
- UNKNOWN_ODSAY_ERROR

주의:
- API key 값 자체를 console.log하지 않는다.
- 사용자 응답에 API key, 전체 요청 URL의 apiKey 값을 포함하지 않는다.
- 서버 로그에도 apiKey는 마스킹한다.
- 원본 에러를 그대로 노출하지 말고 안전하게 요약한다.

4. 안전한 서버 로그 추가
운영 디버깅을 위해 서버 로그에는 원인을 확인할 수 있게 하되, 민감정보는 제거한다.

권장 로그:
- 요청 좌표 요약
- endpoint path
- HTTP status
- ODsay error code/message 요약
- fallback 사용 여부

금지 로그:
- ODSAY_API_KEY 전체 값
- Authorization 또는 apiKey 포함 전체 URL
- 사용자 민감정보

apiKey가 포함된 URL을 로그로 남겨야 한다면 반드시 마스킹한다.
예: apiKey=***MASKED***

5. API 응답 구조 유지
기존 클라이언트가 깨지지 않도록 응답 구조는 크게 바꾸지 않는다.

권장 구조:
{
  "ok": true,
  "mode": "ODSAY_OR_FALLBACK",
  "data": {
    "available": true,
    "route": TransitSummary,
    "congestion": TransitCongestion
  },
  "meta": {
    "source": "ODSAY" | "FALLBACK",
    "fallback": boolean,
    "reason": string | null
  }
}

실제 ODsay 성공 시:
- route.source는 "ODSAY" 또는 기존 타입에 맞는 실제 출처 값
- meta.source는 "ODSAY"
- meta.fallback은 false
- reason은 null 또는 "ODSAY_SUCCESS"

fallback 시:
- route는 기존 fallbackRoute
- meta.source는 "FALLBACK"
- meta.fallback은 true
- reason은 실제 원인 값

6. ODsay 응답 normalize 확인
실제 ODsay 응답을 내부 TransitSummary 형태로 변환하는 로직을 확인한다.

필수 포함:
- totalDurationMin
- transferCount
- steps
- 각 step의 mode
- walk/subway/bus 구분
- distanceM 또는 durationMin
- description

주의:
- ODsay 원본 구조가 예상과 다를 때 throw하지 말고 fallback 처리한다.
- 대전 도시철도/버스 경로가 없을 경우에도 안전하게 fallback 처리한다.
- 혼잡도는 ODsay 실시간 정보가 아니라 기존 “과거 패턴 기반 예측형 혼잡도” 계산/설명을 유지한다.

7. 테스트 작성/보강
실제 ODsay API를 테스트에서 호출하지 않는다.
fetch mock을 사용한다.

필수 테스트:
- origin/destination 좌표가 SX/SY/EX/EY에 올바르게 매핑된다.
- lat/lng가 반대로 들어가지 않는지 확인한다.
- ODSAY_API_KEY가 없으면 안전한 fallback reason을 반환한다.
- HTTP 401/403/500 응답 시 각각 reason을 구분한다.
- ODsay error code/msg가 있으면 reason에 반영한다.
- ODsay error code/msg가 비어 있어도 빈 문자열 reason이 되지 않는다.
- timeout 또는 fetch 실패 시 fallback으로 안전하게 응답한다.
- 실제 ODsay 성공 응답 mock을 TransitSummary로 normalize한다.
- API key가 응답 문자열에 포함되지 않는다.
- 응답 문자열에 금지 표현이 포함되지 않는다.

금지 표현:
- 시연용
- Mock
- MOCK
- 더미
- 사고 확률
- 예측 확률
- probability
- 실시간 혼잡도
- 반드시 반납
- 운전 금지

주의:
- 테스트명에 Mock/시연용/더미 표현을 새로 추가하지 않는다.
- 기존 테스트가 mode를 기대한다면 새 구조에 맞춰 최소 수정한다.
