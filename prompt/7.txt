너는 15년차 Next.js 개발자이자 공공데이터·AI 서비스 운영 QA 전문가다.
나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.

현재 상황:
- 이전 작업 1~24는 완료되었다.
- 실사용 전환 프롬프트 1~6도 진행했거나 진행 중이다.
- Mock/시연용 표현 정리, RecentPlaceList localStorage, Kakao Local 디버깅, ODsay 디버깅, Supabase 저장/조회 확인, Claude 운영 확인까지 점검했다.
- 이제는 새 기능 구현이 아니라 운영 배포 URL에서 전체 사용자 흐름이 실제로 이어지는지 Smoke Test를 진행해야 한다.

이번 작업은 새롭게 시작하는 실사용 전환 프롬프트 7번이다.
파일명 기준으로는 7.txt 작업이다.

이번 작업명:
운영 Smoke Test

작업 목적:
- Vercel Production 배포 환경에서 SilverWay 전체 흐름이 실제로 동작하는지 확인한다.
- 사용자가 /analyze에서 장소를 검색하고, 분석을 실행하고, 결과 페이지에서 저장된 결과를 확인하는 흐름을 검증한다.
- Kakao Local, ODsay, Weather, Claude, Supabase 저장/조회가 각각 실제 응답인지 fallback인지 구분한다.
- fallback이 발생하더라도 화면이 깨지지 않고 사용자가 이해 가능한 결과를 보는지 확인한다.
- 이번 작업은 원칙적으로 코드 수정이 아니라 운영 테스트와 결과 정리다.
- 단, 명백한 오탈자/테스트 명령어 문서 오류 수준의 작은 수정만 허용한다.

절대 하지 말 것:
- 새 기능 구현하지 마.
- 디자인 변경하지 마.
- 레이아웃 변경하지 마.
- Kakao Local 연동을 다시 구현하지 마.
- ODsay 연동을 다시 구현하지 마.
- Weather 연동을 다시 구현하지 마.
- Claude 연동을 다시 구현하지 마.
- Supabase 저장/조회 로직을 다시 구현하지 마.
- Kakao Map SDK를 수정하지 마.
- RecentPlaceList localStorage 로직을 수정하지 마.
- fallback 구조를 제거하지 마.
- 회원가입/auth 구현하지 마.
- 관리자 페이지 구현하지 마.
- 결제/알림/카카오 공유 구현하지 마.
- 패키지 설치하지 마.
- 대규모 리팩토링하지 마.

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 새로 추가하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- fallback은 운영 안정성을 위한 대체 응답으로 유지한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

테스트 대상 운영 URL:
https://silverway.codegenie.co.kr

고정 테스트 시나리오:
- 출발지: 대전광역시청
- 도착지: 충남대학교병원
- 연령대: 70대
- 출발시간: 오늘 오전 10시 또는 현재 UI에서 선택 가능한 기본 시간

Smoke Test 범위:

1. Health Check
다음 API를 호출한다.

curl -s https://silverway.codegenie.co.kr/api/health

확인할 것:
- ok true
- supabaseUrl true
- supabaseAnonKey true
- supabaseServiceRoleKey true 또는 서버 저장 가능 상태
- kakaoRestApiKey true
- kakaoMapKey true
- odsayApiKey true
- weatherApiKey true
- anthropicApiKey true

주의:
- /api/health는 키 값을 노출하면 안 된다.
- boolean 상태만 노출해야 한다.

2. Kakao Local 장소검색 확인
GET과 POST를 모두 테스트한다.

GET:
curl -s "https://silverway.codegenie.co.kr/api/kakao/search?q=%EB%8C%80%EC%A0%84%EA%B4%91%EC%97%AD%EC%8B%9C%EC%B2%AD"

POST:
curl -s -X POST https://silverway.codegenie.co.kr/api/kakao/search \
  -H "Content-Type: application/json" \
  -d '{"query":"대전광역시청","size":3}'

정상 기준:
- ok true
- data 배열 존재
- data[0].name/address/lat/lng 존재
- meta.fallback false
- meta.source가 KAKAO_LOCAL 또는 실제 Kakao 출처
- data[].source가 SAMPLE이 아님

fallback이면 기록할 것:
- meta.reason
- GET/POST 중 어느 방식이 실패하는지
- query가 제대로 전달되는지
- Vercel Function Logs 확인 필요 여부

3. ODsay 대중교통 경로 확인
다음 API를 호출한다.

curl -s -X POST https://silverway.codegenie.co.kr/api/route \
  -H "Content-Type: application/json" \
  -d '{"origin":{"name":"대전광역시청","lat":36.3504,"lng":127.3845},"destination":{"name":"충남대학교병원","lat":36.3166,"lng":127.4156}}'

정상 기준:
- ok true
- data.route 존재
- route.totalDurationMin 존재
- route.transferCount 존재
- route.steps 배열 존재
- meta.fallback false이면 실제 ODsay 성공
- meta.fallback true이면 reason이 구체적으로 나와야 함
- reason이 "ODsay error code= msg="처럼 빈 값이면 미해결로 기록

주의:
- 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도” 표현이어야 한다.

4. Weather API 확인
다음 API를 호출한다.

curl -s https://silverway.codegenie.co.kr/api/weather

또는 현재 route가 POST를 요구하면 프로젝트 코드 기준 명령어로 테스트한다.

정상 기준:
- ok true
- weather 또는 data 존재
- 기상 위험 관련 score/description 존재
- meta.fallback false이면 실제 기상청 API 성공
- meta.fallback true이면 reason이 구체적으로 나와야 함

주의:
- WEATHER_API_KEY는 서버에서만 사용되어야 한다.
- API 응답에 공공데이터 키가 노출되면 안 된다.

5. Claude Report 확인
다음 API를 호출한다.

curl -s -X POST https://silverway.codegenie.co.kr/api/report \
  -H "Content-Type: application/json" \
  -d '{"originName":"대전광역시청","destinationName":"충남대학교병원","departureTime":"2026-05-05T10:00:00+09:00","ageGroup":"70s","drivingRisk":{"score":64,"level":"MEDIUM","label":"보통","description":"공공데이터 기반 사고 패턴, 시간대, 기상 조건, 이동 지역 특성을 조합한 의사결정 보조용 운전 위험 지수입니다.","factors":[]}}'

정상 기준:
- ok true
- data.title 존재
- data.body 존재
- data.familyMessage 존재
- meta.source가 CLAUDE면 실제 Claude 성공
- meta.source가 TEMPLATE이면 fallback 성공
- fallback이면 reason이 구체적으로 나와야 함
- 리포트가 면허 반납을 강요하지 않아야 함

6. /api/analyze 전체 분석 확인
다음 API를 호출한다.

curl -s -X POST https://silverway.codegenie.co.kr/api/analyze \
  -H "Content-Type: application/json" \
  -d '{"origin":{"name":"대전광역시청","address":"대전 서구 둔산로 100","lat":36.3504,"lng":127.3845},"destination":{"name":"충남대학교병원","address":"대전 중구 문화로 282","lat":36.3166,"lng":127.4156},"departureTime":"2026-05-05T10:00:00+09:00","ageGroup":"70s"}'

정상 기준:
- ok true
- data 존재
- data.drivingRisk.score 존재
- data.drivingRisk.description에 “운전 위험 지수” 포함
- data.transit 또는 route 정보 존재
- data.report 존재
- resultId 존재
- persistence.saved true이면 Supabase 저장 성공
- resultId가 mock-* 또는 fallback-* 형식이면 저장 실패 가능성으로 기록
- fallbackFlags가 있으면 어떤 외부 API가 fallback인지 기록
- mode/message에 Mock/시연용 표현이 없어야 함

7. Supabase 저장 확인
/api/analyze 응답에서 받은 resultId를 사용한다.

확인 방법:
- Supabase Dashboard
- Table Editor
- analysis_logs
- resultId와 같은 id의 row가 생겼는지 확인

확인할 것:
- id
- origin/destination 정보
- risk_score 또는 result_json 내부 drivingRisk
- result_json 또는 payload
- created_at

주의:
- 이 단계는 콘솔 확인이 필요하므로 결과에 “확인 완료/미확인”을 적는다.

8. /api/result/[id] 조회 확인
/api/analyze에서 받은 resultId로 호출한다.

curl -s https://silverway.codegenie.co.kr/api/result/{resultId}

정상 기준:
- ok true
- data 존재
- data.id 또는 저장 row id가 resultId와 매칭
- meta.source가 SUPABASE이면 실제 조회 성공
- meta.fallback false이면 정상
- 없는 id 테스트 시에도 화면이 깨지지 않아야 함

없는 id 테스트:
curl -s https://silverway.codegenie.co.kr/api/result/not-existing-result-id

정상 기준:
- 500으로 깨지지 않음
- fallback 또는 안전한 안내 응답
- meta.reason이 RESULT_NOT_FOUND 또는 유사한 값

9. 브라우저 사용자 플로우 확인
브라우저에서 직접 확인한다.

경로:
1. https://silverway.codegenie.co.kr 접속
2. /analyze 이동
3. 출발지 “대전광역시청” 검색
4. 도착지 “충남대학교병원” 검색
5. 장소 선택
6. 연령대/출발시간 선택
7. 분석 실행
8. /result/[id] 이동 확인
9. 결과 페이지 새로고침
10. 결과가 유지되는지 확인

확인할 것:
- 주소검색 결과가 실제 Kakao Local 기반인지
- 최근 장소가 localStorage에 저장되는지
- 분석 중 로딩 UI가 자연스러운지
- 결과 페이지에서 운전 위험 지수 표시가 정상인지
- 대체 경로 표시가 정상인지
- 혼잡도가 “과거 패턴 기반 예측형 혼잡도”로 표시되는지
- 가족 공유용 리포트 문구가 정상인지
- 새로고침 후 결과가 유지되는지
- 모바일 화면에서 버튼/글씨가 너무 작지 않은지

10. Kakao Map 확인
결과 화면 또는 지도 컴포넌트가 있는 화면에서 확인한다.

확인할 것:
- 지도 로드 성공
- 출발지/도착지 마커 표시
- Kakao Map SDK 에러 시 MapFallback 표시
- NEXT_PUBLIC_KAKAO_MAP_KEY만 클라이언트에서 사용
- KAKAO_REST_API_KEY가 클라이언트에 노출되지 않음

11. fallback 강제 확인
가능하면 로컬에서만 환경변수 일부를 비우고 테스트한다.
운영 환경에서는 키를 지우지 않는다.

로컬 확인 예:
- ODSAY_API_KEY 비움 → route fallback 확인
- WEATHER_API_KEY 비움 → weather fallback 확인
- ANTHROPIC_API_KEY 비움 → template report fallback 확인

정상 기준:
- 화면이 깨지지 않음
- fallbackFlags 또는 meta.fallback으로 상태 확인 가능
- 사용자에게 과도한 개발자 에러가 노출되지 않음

12. 표현 원칙 확인
브라우저와 API 응답에서 아래 표현이 보이면 기록한다.

금지 또는 수정 대상:
- 시연용
- Mock
- MOCK
- 더미
- demo
- 사고 확률
- 예측 확률
- probability
- 실시간 혼잡도
- 반드시 반납
- 운전 금지

허용 가능:
- fallback이라는 개발/운영 안정성 용어
- “실제 사고 확률이 아니라” 같은 부정 문맥의 설명은 기존 정책상 허용 가능하나 사용자 화면에서 과도하게 반복되면 문구 개선 후보로 기록

결과 정리 파일:
- ./prompt/smoke-test-result.txt 파일을 생성한다.

- 이번 작업에서 생성하거나 수정해도 되는 파일은 ./prompt/smoke-test-result.txt 하나만 허용한다.
