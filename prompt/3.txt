너는 15년차 Next.js 개발자이자 공공데이터·AI 서비스 실서비스 전환 전문가다.
나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.

현재 상황:
- 이전 작업 1~24는 완료되었다.
- Kakao Local 서버 구조는 이미 구현되어 있다.
- Kakao Map SDK와 Kakao Local REST API 역할도 이미 분리되어 있다.
- Vercel 환경변수 보안 분리도 완료되어 있다.
- 이번 작업은 Kakao Local 연동을 새로 만드는 것이 아니라, 운영 환경에서 fallback으로 떨어지는 원인을 디버깅하고 최소 수정하는 작업이다.

현재 확인된 문제:
배포 URL에서 Kakao Local 검색 테스트 시 아래와 같은 응답이 나온다.

{
  "ok": true,
  "mode": "KAKAO_OR_FALLBACK",
  "data": [
    {
      "name": "대전광역시청",
      "address": "대전 서구 둔산로 100",
      "lat": 36.3504,
      "lng": 127.3845,
      "source": "SAMPLE"
    }
  ],
  "meta": {
    "source": "FALLBACK",
    "fallback": true,
    "reason": "검색어가 너무 짧습니다."
  }
}

중요 판단:
- “대전광역시청”은 짧은 검색어가 아니다.
- 따라서 Kakao API 키 문제라기보다 /api/kakao/search route에서 query/body 파싱이 제대로 안 되거나, GET/POST 입력 형식이 맞지 않는 문제일 가능성이 높다.
- 단, Kakao 개발자 콘솔 설정 문제도 배제하지 않는다.
- KAKAO_REST_API_KEY는 서버에서만 사용해야 한다.

이번 작업은 새롭게 시작하는 실사용 전환 프롬프트 3번이다.
파일명 기준으로는 3.txt 작업이다.

이번 작업명:
Kakao Local 운영 디버깅 및 query 파싱 정리

작업 목적:
- /api/kakao/search가 운영 환경에서 실제 Kakao Local API 결과를 반환하도록 디버깅한다.
- GET 방식과 POST 방식 모두에서 검색어가 정상 파싱되도록 정리한다.
- fallback이 발생할 경우 reason이 실제 원인을 알 수 있게 개선한다.
- Kakao Local 서버 연동 구조는 재작성하지 않고 최소 수정한다.
- 기존 디자인, 레이아웃, UI 스타일은 변경하지 않는다.

절대 하지 말 것:
- Kakao Local 연동을 처음부터 다시 만들지 마.
- Kakao Map SDK를 수정하지 마.
- 지도/마커 기능을 건드리지 마.
- ODsay 연동을 수정하지 마.
- Weather 연동을 수정하지 마.
- Claude 연동을 수정하지 마.
- Supabase 저장/조회 로직을 수정하지 마.
- RecentPlaceList localStorage 로직을 수정하지 마.
- 디자인 변경하지 마.
- 레이아웃 변경하지 마.
- 패키지 설치하지 마.
- 대규모 리팩토링하지 마.
- 클라이언트에 KAKAO_REST_API_KEY를 노출하지 마.
- NEXT_PUBLIC_KAKAO_REST_API_KEY를 만들지 마.

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 새로 추가하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- fallback은 운영 안정성을 위한 대체 응답으로 유지한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

점검 및 수정 지침:

1. /api/kakao/search route 입력 파싱 확인
현재 route가 GET만 지원하는지, POST만 지원하는지, 둘 다 지원하는지 확인한다.

반드시 지원할 입력:
- GET /api/kakao/search?q=대전광역시청
- GET /api/kakao/search?query=대전광역시청
- POST /api/kakao/search
  body: { "query": "대전광역시청", "size": 3 }
- POST /api/kakao/search
  body: { "q": "대전광역시청", "size": 3 }

권장:
- GET과 POST를 모두 지원한다.
- q와 query 둘 다 허용한다.
- 검색어 trim 처리한다.
- size는 기본 5, 최대 10 정도로 제한한다.
- query가 2글자 미만이면 fallback sample을 반환하기보다 ok true + 빈 배열 또는 명확한 안내를 반환한다.
- 단, 기존 UI가 fallback 데이터를 기대하고 있다면 깨지지 않도록 현재 응답 구조를 유지한다.

2. 현재 문제 원인 확인
“검색어가 너무 짧습니다.”가 왜 발생하는지 코드에서 확인한다.

확인할 것:
- POST body의 query를 읽고 있는가?
- GET searchParams의 q를 읽고 있는가?
- GET searchParams의 query를 읽고 있는가?
- route가 GET 요청인데 테스트가 POST로 되어 있거나 반대는 아닌가?
- body를 한 번 읽은 뒤 다시 읽는 문제는 없는가?
- query 값이 undefined/null/empty string으로 전달되는 경로가 있는가?
- 한글 URL 인코딩 문제는 없는가?
- 클라이언트에서 q가 아니라 keyword 같은 다른 이름으로 보내고 있지는 않은가?

3. lib/kakao/searchPlace.ts 확인
기존 searchPlace 함수는 유지하되, 아래를 점검한다.

확인할 것:
- KAKAO_REST_API_KEY를 process.env.KAKAO_REST_API_KEY에서 읽는가?
- 서버 전용 파일에서만 사용되는가?
- Authorization header가 정확한가?
  Authorization: KakaoAK ${KAKAO_REST_API_KEY}
- Kakao keyword search endpoint가 정확한가?
  https://dapi.kakao.com/v2/local/search/keyword.json
- 주소 fallback을 쓴다면 address endpoint가 정확한가?
  https://dapi.kakao.com/v2/local/search/address.json
- Kakao 응답의 x/y를 lng/lat으로 정확히 변환하는가?
  x = lng
  y = lat
- Kakao API 오류 발생 시 원본 status, errorType, message 일부를 서버 로그 또는 meta.reason에 안전하게 남기는가?

주의:
- API key 값 자체를 console.log하지 않는다.
- 사용자 응답에 API key, Authorization header를 포함하지 않는다.
- 원본 에러를 그대로 노출하지 말고 안전하게 요약한다.

4. fallback reason 개선
현재 reason이 항상 “검색어가 너무 짧습니다.”로 나오는 문제가 있다면 실제 원인에 맞게 구분한다.

권장 reason:
- EMPTY_QUERY
- KAKAO_REST_API_KEY_MISSING
- KAKAO_API_HTTP_ERROR
- KAKAO_API_EMPTY_RESULT
- KAKAO_API_TIMEOUT
- KAKAO_API_PARSE_ERROR
- UNKNOWN_KAKAO_ERROR

사용자-facing message 예시:
- "검색어를 입력해 주세요."
- "장소 검색 결과가 없습니다."
- "장소 검색을 일시적으로 사용할 수 없습니다."

meta.reason에는 개발자가 원인을 파악할 수 있는 짧은 값 또는 문장을 넣는다.
단, API key나 민감정보는 절대 포함하지 않는다.

5. API 응답 구조 유지
기존 클라이언트가 깨지지 않도록 응답 구조는 크게 바꾸지 않는다.

권장 구조:
{
  "ok": true,
  "mode": "KAKAO_OR_FALLBACK",
  "data": Place[],
  "meta": {
    "source": "KAKAO_LOCAL" | "FALLBACK",
    "fallback": boolean,
    "reason": string | null
  }
}

실제 Kakao 성공 시:
- data[].source는 "KAKAO_LOCAL" 또는 기존 타입에 맞는 실제 출처 값
- meta.source는 "KAKAO_LOCAL"
- meta.fallback은 false
- reason은 null 또는 "KAKAO_LOCAL_SUCCESS"

fallback 시:
- data는 기존 fallback 장소 또는 빈 배열
- meta.source는 "FALLBACK"
- meta.fallback은 true
- reason은 실제 원인 값

6. 테스트 작성/보강
실제 Kakao API를 테스트에서 호출하지 않는다.
fetch mock을 사용한다.

필수 테스트:
- GET q 파라미터로 검색어를 전달하면 searchPlace에 정상 전달된다.
- GET query 파라미터로 검색어를 전달하면 searchPlace에 정상 전달된다.
- POST body query로 검색어를 전달하면 searchPlace에 정상 전달된다.
- POST body q로 검색어를 전달하면 searchPlace에 정상 전달된다.
- 검색어가 비어 있으면 적절한 reason을 반환한다.
- Kakao 응답의 x/y가 lng/lat으로 변환된다.
- Kakao API 오류 시 fallback으로 안전하게 응답한다.
- API key 값이 응답에 포함되지 않는다.
- 응답 문자열에 금지 표현이 포함되지 않는다.

금지 표현:
- 시연용
- Mock
- MOCK
- 더미
- 사고 확률
- 예측 확률
- probability
- 실시간 혼잡도
- 반드시 반납
- 운전 금지

주의:
- 테스트명에 Mock/시연용/더미 표현을 새로 추가하지 않는다.
- 기존 테스트가 mode를 기대한다면 새 구조에 맞춰 최소 수정한다.
