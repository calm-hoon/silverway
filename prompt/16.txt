너는 15년차 Next.js 개발자이자 공공데이터·AI 서비스 MVP 설계 전문가이며, Supabase PostgreSQL 기반 MVP 백엔드 구조 설계에 익숙한 풀스택 개발자다.
나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.
목표는 공모전 시연용 MVP를 넘어 실제 배포 가능한 앱으로 확장 가능한 구조를 만드는 것이다.

중요:

- 기존 UI 디자인은 가능한 유지한다.

- 아직 실제 Kakao Local API, Kakao Map, ODsay, Weather, Claude API 연동은 하지 않는다.
- 아직 회원가입/auth 구현하지 않는다.
- 아직 관리자 페이지, 결제, 알림, 카카오 공유 구현하지 않는다.

- Supabase 환경변수가 없거나 DB 연결에 실패해도 sampleAnalysis fallback으로 화면이 깨지지 않게 한다.

- 이번 작업은 디자인 변경/반영 단계가 아니다.

# 🚨 디자인 변경/벤치마킹 반영 단계입니다

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 사용하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- API 실패 시 sampleAnalysis, sampleRoute, sampleWeather, generateTemplateReport 등 fallback으로 화면이 깨지지 않게 한다.
- “무장애”보다 “이동권 보장”, “이동 편의 동선” 표현을 우선 사용한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

작업 목표:
- /api/analyze에서 생성된 Mock 분석 결과를 Supabase analysis_logs 테이블에 저장한다.
- 저장 성공 시 resultId를 반환하고, 입력 화면에서 /result/[id]로 이동할 수 있게 한다.
- /api/result/[id]에서 Supabase analysis_logs를 조회한다.
- 조회 실패, id 없음, 환경변수 없음, DB 연결 실패 시 sampleAnalysis fallback을 반환한다.
- 실제 외부 API 연동 없이도 “분석 → 저장 → 결과 페이지 조회” 흐름이 동작하도록 한다.
- service role key는 서버 전용으로만 사용하고 클라이언트에 노출하지 않는다.

구현 지침:

1. lib/supabase/analysisLogs.ts
- analysis_logs 저장/조회 전용 서버 유틸을 만든다.
- 이 파일은 서버에서만 사용된다는 주석을 남긴다.
- service role 기반 Supabase client를 사용하되, 클라이언트 컴포넌트에서 import되지 않게 한다.
- 실제 query는 이 파일 안에만 모은다.

권장 함수:
export async function saveAnalysisLog(result: AnalysisResult): Promise<SaveAnalysisLogResult>
export async function getAnalysisLogById(id: string): Promise<GetAnalysisLogResult>

권장 반환 타입:
type SaveAnalysisLogResult =
  | { ok: true; id: string; result: AnalysisResult }
  | { ok: false; reason: string; fallback: AnalysisResult };

type GetAnalysisLogResult =
  | { ok: true; result: AnalysisResult; source: "SUPABASE" }
  | { ok: false; reason: string; fallback: AnalysisResult; source: "FALLBACK" };

- Supabase 환경변수가 없으면 throw하지 말고 ok false와 fallback을 반환한다.
- insert 실패나 select 실패도 ok false와 fallback을 반환한다.
- result JSON은 analysis_logs.result 또는 result_json 같은 JSONB 컬럼에 저장한다.
- 작업 2 migration의 실제 컬럼명에 맞춰 작성한다.
- 컬럼명이 다를 수 있으면 README에 migration과 컬럼명을 확인하라고 남긴다.
- id는 DB에서 uuid default가 있으면 DB 값을 사용한다.
- DB id를 얻기 어렵다면 result.id를 저장하고 select로 반환한다.
- updated_at 같은 컬럼이 없다면 사용하지 않는다.

주의:
- service role key는 서버 전용이다.
- NEXT_PUBLIC_ 접두사를 붙이지 않는다.
- 클라이언트 컴포넌트에서 saveAnalysisLog/getAnalysisLogById를 직접 import하지 않는다.
- 실제 외부 API 호출은 하지 않는다.

2. app/api/analyze/route.ts
- 기존 Mock 분석 생성 로직은 유지한다.
- POST /api/analyze 호출 시 Mock AnalysisResult를 만든다.
- 이후 saveAnalysisLog(result)를 호출해 Supabase 저장을 시도한다.
- 저장 성공 시 응답 data.id 또는 resultId에 저장된 id를 반영한다.
- 저장 실패 시에도 200 JSON을 반환하고, sampleAnalysis 기반 fallback 또는 Mock result를 반환한다.
- 응답에는 저장 상태를 알 수 있는 meta를 포함한다.

권장 응답 구조:
{
  "ok": true,
  "mode": "MOCK_WITH_STORAGE",
  "data": AnalysisResult,
  "resultId": "uuid-or-mock-id",
  "message": "Mock 분석 결과를 반환했습니다.",
  "meta": {
    "stored": true,
    "storageSource": "SUPABASE"
  },
  "fallbackFlags": {
    "analysis": true,
    "route": true,
    "weather": true,
    "report": true
  }
}

저장 실패 시:
{
  "ok": true,
  "mode": "MOCK_WITH_STORAGE_FALLBACK",
  "data": AnalysisResult,
  "resultId": "test",
  "message": "저장 연결이 없어 fallback 결과를 반환했습니다.",
  "meta": {
    "stored": false,
    "storageSource": "FALLBACK",
    "reason": "..."
  }
}

주의:
- 저장 실패를 500으로 처리하지 않는다.
- 화면이 깨지지 않도록 200 fallback 응답을 우선한다.
- 실제 Kakao/ODsay/Weather/Claude 호출은 하지 않는다.

3. app/api/result/[id]/route.ts
- GET /api/result/[id]에서 getAnalysisLogById(id)를 호출한다.
- 조회 성공 시 Supabase에 저장된 AnalysisResult를 반환한다.
- 조회 실패 시 sampleAnalysis fallback을 반환한다.
- 없는 id라도 현재 MVP 단계에서는 404가 아니라 200 fallback을 반환한다.
- 응답 meta에 requestedId, source, fallback 여부를 포함한다.

권장 응답 구조:
{
  "ok": true,
  "mode": "RESULT_LOOKUP",
  "data": AnalysisResult,
  "message": "결과를 반환했습니다.",
  "meta": {
    "requestedId": "abc123",
    "source": "SUPABASE",
    "fallback": false
  }
}

fallback 시:
{
  "ok": true,
  "mode": "RESULT_LOOKUP_FALLBACK",
  "data": AnalysisResult,
  "message": "저장된 결과를 찾지 못해 예시 결과를 반환했습니다.",
  "meta": {
    "requestedId": "abc123",
    "source": "FALLBACK",
    "fallback": true
  }
}

- UI 디자인은 기존 스타일을 유지하고, 큰 디자인 변경은 하지 않는다.

주의:

- 클라이언트에서 service role key 사용하지 않는다.
- 클라이언트에서는 /api/analyze만 호출한다.

7. 테스트
Supabase 실제 연결 테스트는 하지 않는다.
환경변수와 DB에 의존하지 않는 단위 테스트 중심으로 작성한다.

권장:
- Supabase client를 직접 mock 하기 어렵다면, 저장/조회 결과를 만드는 순수 helper를 분리해 테스트한다.
- 또는 환경변수가 없을 때 fallback 결과를 반환하는 케이스를 테스트한다.

필수 테스트 케이스:
- 환경변수가 없거나 client 생성이 불가능해도 saveAnalysisLog가 throw하지 않는다.
- getAnalysisLogById에 빈 id가 들어와도 throw하지 않는다.
- fallback result에 “운전 위험 지수” 표현이 포함된다.
- fallback result의 혼잡도 설명에 “과거 패턴 기반 예측형 혼잡도” 표현이 포함된다.
- 반환 문자열 전체에 금지 표현이 포함되지 않는다.

금지 표현:
- 사고 확률
- 예측 확률
- probability
- 실시간 혼잡도
- 운전 금지
- 반드시 반납

테스트명에도 금지 표현을 쓰지 않는다.
