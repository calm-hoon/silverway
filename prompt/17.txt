나는 2026년 대전광역시 공공데이터·AI 활용 창업경진대회에 제출할 SilverWay 서비스를 개발하고 있다.

서비스명은 SilverWay이고, 한 줄 정의는 “AI 기반 고령자 이동 및 면허 반납 의사결정 지원 서비스”다.
목표는 공모전 시연용 MVP를 넘어 실제 배포 가능한 앱으로 확장 가능한 구조를 만드는 것이다.

중요:
- 이번 작업은 디자인 변경/벤치마킹 반영 단계가 아니다.
- 기존 UI 디자인은 가능한 유지한다.
- 이번 작업은 실제 대중교통 대체 경로 조회를 위한 ODsay API 연동 단계다.
- 아직 Kakao Local API, Kakao Map, Weather, Claude API 연동은 하지 않는다.
- 아직 회원가입/auth 구현하지 않는다.
- 아직 관리자 페이지, 결제, 알림, 카카오 공유 구현하지 않는다.
- ODsay API가 실패하거나 키가 없거나 응답 구조가 예상과 달라도 sampleRoute fallback으로 화면이 깨지지 않게 한다.
- ODsay API key는 서버 Route Handler에서만 사용하고 클라이언트에 노출하지 않는다.

디자인 관련 주의:
- 이번 작업은 디자인 변경/반영 단계가 아니다.
- 디자인이나 벤치마킹 사이트 반영이 필요한 작업이 다시 오면, 반드시 작업 시작 전에 아래처럼 큰 제목으로 먼저 알려줘.

# 🚨 디자인 변경/벤치마킹 반영 단계입니다

서비스 표현 원칙:
- 위험도 점수는 실제 사고 확률이 아니라 “운전 위험 지수”다.
- “사고 확률”, “예측 확률”, “probability” 표현은 사용하지 않는다.
- AFC 혼잡도는 실시간이 아니라 “과거 패턴 기반 예측형 혼잡도”로 표현한다.
- API 실패 시 sampleAnalysis, sampleRoute, sampleWeather, generateTemplateReport 등 fallback으로 화면이 깨지지 않게 한다.
- “무장애”보다 “이동권 보장”, “이동 편의 동선” 표현을 우선 사용한다.
- 면허 반납은 강요가 아니라 가족과 함께 논의할 수 있는 의사결정 보조 관점으로 표현한다.

작업 목표:
- ODsay API를 서버 측에서 호출하는 연동 구조를 만든다.
- 출발지/도착지 좌표를 받아 대중교통 대체 경로를 조회한다.
- ODsay 원본 응답을 앱 내부 TransitSummary 타입으로 정규화한다.
- /api/route Route Handler에서 실제 ODsay 조회를 시도하고, 실패 시 sampleRoute fallback을 반환한다.
- /api/analyze에서 가능한 경우 ODsay 대중교통 경로 결과를 활용한다.
- API key 누락, 네트워크 실패, 응답 오류, 경로 없음 상황에서도 앱이 깨지지 않게 한다.
- 실제 Kakao 좌표 검색은 아직 하지 않고, 입력 화면의 mock 장소 좌표 또는 request body 좌표를 사용한다.

구현 지침:

1. 환경변수
- .env.local.example에 ODSAY_API_KEY가 이미 있는지 확인한다.
- 없으면 추가한다.

ODsay API key:
ODSAY_API_KEY=

주의:
- NEXT_PUBLIC_ 접두사를 붙이지 않는다.
- 클라이언트 컴포넌트에서 사용하지 않는다.
- 서버 Route Handler 또는 서버 유틸에서만 읽는다.

2. lib/odsay/types.ts
- ODsay API 원본 응답 타입은 필요한 최소 필드만 정의한다.
- 전체 원본 스펙을 모두 타입화하려고 하지 않는다.
- unknown을 받은 뒤 안전하게 파싱하는 구조를 우선한다.
- any 사용은 피하되, 원본 응답이 복잡하면 최소 범위에서만 사용한다.

권장 타입:
export type OdsayRouteRequest = {
  originLat: number;
  originLng: number;
  destinationLat: number;
  destinationLng: number;
};

export type OdsayRouteResult =
  | { ok: true; transit: TransitSummary; source: "ODSAY" }
  | { ok: false; transit: TransitSummary; source: "FALLBACK"; reason: string };

필요하면 OdsayRawResponse, OdsayPath, OdsaySubPath 같은 내부 타입을 최소 정의한다.

3. lib/odsay/normalizeOdsayRoute.ts
- ODsay 원본 응답을 앱 내부 TransitSummary 타입으로 변환한다.
- 순수 함수로 작성한다.
- API 호출은 하지 않는다.
- 테스트하기 쉬운 구조로 만든다.

권장 함수:
export function normalizeOdsayRoute(raw: unknown): TransitSummary | null

정규화 기준:
- 전체 소요 시간 → durationMin
- 환승 횟수 → transferCount
- 도보 시간 → walkingTimeMin
- 경로 단계 → steps
- 지하철/버스/도보 구간을 TransitStep으로 변환
- 혼잡도는 ODsay가 아니라 AFC 기반이므로 여기서 실시간 혼잡도처럼 만들지 않는다.
- transit.congestion이 필요하면 기존 sampleRoute.congestion 또는 calculateCongestion fallback을 사용한다.
- 원본 응답에 경로가 없거나 파싱 실패 시 null을 반환한다.

주의:
- “실시간 혼잡도” 표현 금지.
- ODsay 응답을 과도하게 신뢰하지 말고 필요한 필드만 안전하게 추출한다.
- 응답 구조가 바뀌어도 앱이 throw하지 않도록 optional chaining과 타입 가드를 사용한다.

4. lib/odsay/getTransitRoute.ts
- ODSAY_API_KEY를 서버 측에서 읽는다.
- ODsay API를 호출해 대중교통 경로를 조회한다.
- fetch timeout 또는 AbortController를 사용해 너무 오래 대기하지 않게 한다.
- 실패 시 sampleRoute fallback을 반환한다.
- 외부 API 호출 실패를 throw로 전파하지 말고 OdsayRouteResult 형태로 반환한다.

권장 함수:
export async function getTransitRoute(input: OdsayRouteRequest): Promise<OdsayRouteResult>

처리해야 할 실패 케이스:
- ODSAY_API_KEY 누락
- 좌표 누락 또는 숫자 아님
- fetch 실패
- HTTP 응답 실패
- ODsay 응답 내 오류
- 경로 없음
- 정규화 실패

fallback 시:
- sampleRoute를 반환한다.
- source는 "FALLBACK"으로 둔다.
- reason에 내부 확인용 짧은 이유를 남긴다.
- 사용자-facing 문구에는 과도한 오류 표현을 쓰지 않는다.

주의:
- ODSAY_API_KEY를 응답 JSON에 포함하지 않는다.
- console.log로 key를 출력하지 않는다.
- 클라이언트에서 직접 ODsay API 호출하지 않는다.

5. lib/odsay/index.ts

예시:
export { getTransitRoute } from "./getTransitRoute";
export { normalizeOdsayRoute } from "./normalizeOdsayRoute";
export type { OdsayRouteRequest, OdsayRouteResult } from "./types";

6. app/api/route/route.ts
- 기존 sampleRoute 반환 route를 실제 ODsay 연동 route로 확장한다.
- GET과 POST 중 하나를 선택한다.
- 추천:
  - POST /api/route
  - body에 origin/destination 좌표를 받는다.
- 기존 GET이 있다면 GET은 sampleRoute health/mock 응답으로 유지해도 된다.

POST body 예시:
{
  "origin": {
    "name": "대전광역시청",
    "lat": 36.3504,
    "lng": 127.3845
  },
  "destination": {
    "name": "충남대학교병원",
    "lat": 36.3166,
    "lng": 127.4156
  }
}

응답 예시:
{
  "ok": true,
  "mode": "ODSAY_OR_FALLBACK",
  "data": TransitSummary,
  "meta": {
    "source": "ODSAY",
    "fallback": false
  }
}

fallback 응답 예시:
{
  "ok": true,
  "mode": "ODSAY_OR_FALLBACK",
  "data": TransitSummary,
  "meta": {
    "source": "FALLBACK",
    "fallback": true,
    "reason": "ODsay API key is missing"
  }
}

주의:
- 실패해도 500보다 200 fallback을 우선한다.
- 정말 잘못된 method만 405 또는 적절한 응답을 사용할 수 있다.
- 사용자 화면이 깨지지 않게 한다.

7. app/api/analyze/route.ts
- 기존 Mock 분석 흐름을 크게 바꾸지 않는다.
- request body에 origin/destination 좌표가 있으면 getTransitRoute를 호출해 transit 값을 반영한다.
- ODsay 실패 시 sampleRoute fallback을 유지한다.
- 저장/조회 구조가 이미 있다면 기존 Supabase 저장 흐름을 깨지 않는다.
- ODsay 결과가 있으면 AnalysisResult.transit에 반영한다.
- fallbackFlags.route를 실제 source에 따라 업데이트한다.
- 다른 API(Kakao, Weather, Claude)는 아직 호출하지 않는다.

9. 테스트
실제 ODsay API를 호출하는 테스트는 작성하지 않는다.
정규화 함수와 fallback 동작만 테스트한다.

tests/odsay/normalizeOdsayRoute.test.ts 필수 테스트:
- 최소 ODsay-like raw 응답을 TransitSummary로 변환한다.
- 경로가 없으면 null을 반환한다.
- 잘못된 raw 입력에서도 throw하지 않는다.
- 정규화 결과에 durationMin, transferCount, steps가 포함된다.
- 문자열 전체에 금지 표현이 포함되지 않는다.

tests/odsay/getTransitRoute-fallback.test.ts 필수 테스트:
- ODSAY_API_KEY가 없을 때 fallback result를 반환한다.
- 잘못된 좌표 입력에서도 throw하지 않는다.
- fallback transit description 또는 congestion description에 “과거 패턴 기반 예측형 혼잡도” 표현이 유지된다.
- 응답 문자열에 key가 포함되지 않는다.
- 금지 표현이 포함되지 않는다.

금지 표현:
- 사고 확률
- 예측 확률
- probability
- 실시간 혼잡도
- realtime
- live congestion
- 운전 금지
- 반드시 반납

테스트명에도 금지 표현을 쓰지 않는다.
